# System Design

The platform is organized around role-based APIs for customers, admins, and delivery agents. Customers create confirmed orders only after requesting a quote, while admins configure the operational data that makes quote generation dynamic: zones, rate cards, agent coverage, and status overrides.

The rate engine is intentionally data-driven. A `Zone` stores area keywords, pincodes, and an optional center coordinate. During quote creation, the service normalizes pickup and drop addresses, extracts a pincode when present, and matches first by configured pincode or by area/zone name contained in the address. After detecting both zones, it looks up an active `RateCard` by `orderType`, pickup zone, and drop zone. This supports intra-zone and inter-zone pricing without hardcoded rates. The system calculates volumetric weight as `L × B × H ÷ 5000`, compares it with actual weight, bills the higher value, applies the configured per-kg rate and minimum charge, and adds the configured COD surcharge only for COD orders. The quote endpoint exposes these components before the customer confirms.

Agent assignment supports both manual dispatch and automatic selection. Admins can directly assign a delivery agent to an order. Auto-assignment filters agents by `deliveryAgent` role, availability, no active order, and either a matching assigned zone or no zone restriction. It then prefers same-zone agents and sorts by Haversine distance between the agent’s latest location and the pickup zone center. Once assigned, the agent is marked unavailable and linked to the active order.

Order state changes are handled through a lifecycle service. Valid statuses include Created, Picked Up, In Transit, Out for Delivery, Delivered, and Failed. Each change updates the order’s current status and appends a new `TrackingHistory` document containing status, actor, note, and timestamp. Existing tracking documents are never edited, preserving a reliable audit trail. Customer order detail returns both the order and its full timeline.

Failed delivery is treated as a recoverable workflow. When an agent or admin marks an order Failed, the reason is stored, the customer is notified, and the agent is released. The customer can submit a new reschedule date. The order attempt count increments, status moves back to Created, and the system immediately tries to auto-assign a new available agent for the next attempt.

Notifications are abstracted behind a service that persists every email/SMS attempt. If webhook URLs are configured, the app posts notification payloads to the selected free-tier provider endpoint with an optional bearer token. If no provider is configured, it falls back to console delivery while still recording the notification status and provider response. This keeps SendGrid, Mailgun, Twilio, or other providers replaceable without changing controller logic.
